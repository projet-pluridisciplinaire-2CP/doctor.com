========================================================================
4. ARCHITECTURE COMMUNE AUX 3 AGENTS
========================================================================

4.1. Entree

Les 3 agents sont des routes protegees:
- un utilisateur connecte est requis
- le backend relit ensuite l'utilisateur metier depuis la DB

Ce pattern garantit:
- pas d'appel anonyme
- pas d'utilisation d'un suivi / patient qui n'appartient pas a
  l'utilisateur connecte


4.2. Providers AI supportes

Les 3 modules supportent la meme famille de providers:

- openrouter
- together
- mistral
- google-ai-studio (Gemini)

Selection:
- si AI_PROVIDER est force dans apps/server/.env, ce provider est utilise
- sinon fallback interne:
  openrouter -> together -> mistral -> google-ai-studio


4.3. Strategie generale de robustesse

Les 3 modules suivent cette strategie:

  Input valide
      ->
  Recuperation du contexte local
      ->
  Logique backend / retrieval / filtrage
      ->
  Appel LLM si utile
      ->
  Parsing JSON
      ->
  Normalisation
      ->
  Validation Zod
      ->
  Retour final protege

Important:

- le backend ne fait pas confiance aveuglement au modele
- le modele n'est pas la source de verite
- la DB locale reste la source verifiable


4.4. Types de garde-fous utilises

Garde-fous techniques:
- timeout provider
- mapping des erreurs HTTP / quota / auth
- parsing JSON robuste
- nettoyage des fences markdown
- validation schema de sortie

Garde-fous metier:
- verification de l'utilisateur
- verification d'appartenance du suivi
- exclusion de medicaments hors shortlist
- refus des questions necessitant un contexte patient dans le chat catalogue
- fallback local quand le LLM est vide ou non fiable

Le module ne depend pas uniquement du modele pour la securite.

Des motifs regex detectent explicitement:

- syncope
- convulsions
- saignement / hemoptysie
- confusion / alteration neurologique

Ces signaux alimentent:
- les warnings
- la recommendation_readiness

Le provider recoit un prompt structure contenant:
- contexte patient
- consultation courante
- historique utile
- traitements
- antecedents

Le modele doit retourner un JSON structure avec:
- recommendation_readiness
- chief_problem

- red_flags[]
- caution_notes[]
- global_missing_information[]


5.7. Normalisation et validation

Apres retour du modele:

1. parsing JSON
2. normalisation:
   - nettoyage des strings
   - fallback de valeurs manquantes
   - normalisation de confidence
   - bornage des tableaux
3. validation Zod stricte

Correction importante ajoutee:
- les champs texte trop longs sont maintenant tronques avant validation
- cela evite les erreurs du type:
  caution_notes.0 too big

Impact:
- Gemini peut repondre plus librement
- le backend garde une sortie valide


5.8. Place reelle du LLM

Dans ce module, le LLM reste central pour:
- la synthese clinique
- les hypotheses
- la formulation

Mais le backend garde la maitrise sur:
- le contexte fourni
- les red flags deterministes
- la validation finale


5.9. Niveau de maturite

Etat actuel:
- techniquement valide
- fonctionnellement utilisable
- le plus stable des 3 agents a ce stade

6. AGENT 2 - ORDONNANCE RECOMMENDATION
========================================================================

6.1. Role fonctionnel

But:
- generer un brouillon d'ordonnance recommande
- a partir d'un contexte patient reel et d'une shortlist medicamenteuse locale

Endpoint:
- POST /trpc/ai.ordonnanceRecommendation.generate

Router:
- /Users/mac/Desktop/doctor.com/packages/api/src/modules/ai/ordonnance-recommendation/router.ts

Input:
- suivi_id
- examen_id optionnel
- include_historical_context
- max_historical_suivis
- max_historical_treatments
- clinical_problem_override optionnel

Output principal:
- provider
- model
- source
- clinical_problem_basis
- candidate_summary
- recommendations
- excluded_candidates
- global_warnings
- disclaimer


6.2. Intention produit

Ce module n'est pas un prescripteur autonome.

Il est concu comme:
- un assistant de brouillon d'ordonnance
- base sur la DB locale medicaments
- avec verification backend
- et validation humaine finale par le medecin


6.3. Pipeline runtime

Schema global:

  Requete frontend
      ->
  Validation input Zod
      ->
  Resolve session / utilisateur
      ->
  Resolve suivi
      ->
  Resolve examen
      ->
  Resolve patient
      ->
  Build clinical context
      ->
  Resolve clinical problem basis
      ->
  Retrieve medication candidates
      ->
  Filter + score candidates
      ->
  LLM draft (si utile)
      ->
  Post-validation
      ->
  Fallback deterministe si besoin
      ->
  Return result


6.4. Clinical problem basis

Le module ne doit pas "deviner" librement la pathologie.

Ordre de priorite implemente:

1. clinical_problem_override
2. suivi.hypothese_diagnostic
3. hypotheseDiagnostic.generate en interne
4. fallback sur motif / conclusion / description_consultation

Impact:
- on ne part pas d'un prompt vide
- on essaie toujours de fonder la recommandation sur un probleme clinique
  explicite


6.5. Contexte clinique

Le contexte clinique reconstruit est proche de celui de hypotheseDiagnostic:
- patient
- contexte femme
- consultation courante
- examen courant
- antecedents
- traitements
- historique
- voyages
- vaccinations
- red flags deterministes

La difference:
- ici, ce contexte sert a filtrer et evaluer des candidats medicamenteux


6.6. Retrieval medicaments

Le module utilise:
- medicamentsService.rechercherMedicaments(...)
- medicamentsService.getMedicamentById(...)

Strategies de retrieval:
- query
- indication
- nom_substance

Le service construit des termes de retrieval a partir:
- chief_problem
- hypotheses
- motif
- hypothese_diagnostic
- conclusion examen
- description_consultation
- expansions metier backend ajoutees

Correction ajoutee:
- expansion specifique selon un profil clinique backend
  exemple:
  antipyretic_simple -> antipyretique / fievre / paracetamol / doliprane


6.7. Profils metier backend

Un profil metier est derive du probleme clinique:

- antipyretic_simple
- analgesic_simple
- bronchodilator_inhaled
- antibiotic_general
- generic

Ce profil sert a:
- enrichir le retrieval
- scorer les candidats
- poser un gate clinique
- construire un fallback deterministe


6.8. Filtrage de securite backend

Ce module contient un filtrage dur AVANT la generation finale.

Exclusions automatiques:

1. Allergie / hypersensibilite
- tokens extraits depuis les signaux d'allergie du patient
- comparaison avec nom + DCI + substances + corpus de securite du medicament

2. Interactions avec traitements actifs
- tokens extraits depuis les traitements actifs
- comparaison avec le corpus interactions du candidat

Si exclusion:
- le medicament sort de la shortlist
- il va dans excluded_candidates avec une raison explicite


6.9. Warnings backend

Pour les candidats non exclus, le backend ajoute des warnings si:
- antecedents du patient a confronter avec precautions / CI
- pas de posologie adulte exploitable
- pas de presentation exploitable
- produit repere comme supprime


6.10. Scoring clinique backend

Le score final ne vient pas du LLM.

Le backend score selon:
- retrieval_score initial
- match du probleme clinique avec indications
- posologie adulte presente
- presentations presentes
- policy scoring specifique au profil
- penalites si warnings
- penalite si produit supprime

Exemples de policy scoring:

antipyretic_simple:
- bonus fort pour paracetamol
- bonus pour indication fievre / antipyretique
- bonus pour monotherapie
- malus pour aspirine
- malus pour AINS non necessaires

bronchodilator_inhaled:
- bonus bronchodilatateur
- bonus forme inalee
- malus forme non inalee

antibiotic_general:
- bonus antibiotique
- malus hors classe


6.11. Gate clinique

Le gate clinique est un filtre "must-match".

Exemple:

Pour antipyretic_simple, un candidat doit matcher:
- paracetamol-like
OU
- indication / classe liee a la fievre

Sinon:
- il est exclu
- meme si son retrieval lexical etait correct

Pourquoi c'est important:
- eviter les faux positifs absurdes
- ne pas laisser survivre un medicament hors sujet juste parce qu'il a
  une posologie adulte ou une monotherapie


6.12. LLM dans ordonnanceRecommendation

Le LLM recoit:
- le probleme clinique
- le contexte compact
- la shortlist validee

Le prompt a ete compactee pour Gemini:
- moins d'historique
- shortlist reduite
- champs tronques
- JSON compacte

Correction provider importante:
- Gemini refusait initialement le responseJsonSchema complexe
- il a donc ete retire uniquement sur cette route pour Gemini
- le backend garde la validation locale derriere


6.13. Post-validation

Apres sortie du modele:
- parse JSON
- normalisation
- validation Zod
- verification que chaque medicament retourne appartient a la shortlist
- refus des doublons dans une meme recommandation

Le modele ne peut donc pas inventer librement:
- ni un medicament absent
- ni un id arbitraire


6.14. Fallback deterministe backend

Changement majeur ajoute:

Si:
- le modele echoue
OU
- retourne recommendations: []

Alors:
- le backend peut construire un brouillon minimal local

Ce fallback:
- choisit le meilleur candidat selon le profil
- exige une posologie adulte exploitable
- construit 1 recommandation locale
- remplit:
  rank
  label
  rationale
  warnings
  ordonnance_draft
  justification

Exemples:
- option antalgique / antipyretique simple
- option bronchodilatatrice inalee
- option antibiotique de reference


6.15. Limite actuelle

Ce module est le plus difficile des 3.

Pourquoi:
- il doit faire du raisonnement therapeutique
- il depend beaucoup de la qualite du retrieval
- il reste sensible a la qualite de la DB scrapee
- les LLM generalistes sont inegaux sur ce genre de sortie

Etat actuel:
- techniquement branche
- beaucoup plus robuste qu'au debut
- mais encore plus experimental que hypotheseDiagnostic

7.10. Candidate features backend

Pour chaque candidat, le service calcule un grand set de features:

- nombre de substances actives
- monotherapie / combinaison
- statut supprime
- association type rhume
- aspirin-like
- nsaid-like
- paracetamol-like
- antipyretic-like
- analgesic-like
- indication fievre
- indication douleur
- toux grasse
- toux seche
- congestion nasale
- bronchodilatateur
- forme inalee
- antibiotique
- posologie adulte disponible
- contenu securite disponible

Ces features permettent un ranking metier deterministe.


7.11. Scoring backend

Le score d'un candidat combine:
- matches lexicaux sur champs riches
- bonus de posologie adulte
- bonus si monotherapie
- bonus de profil metier
- malus si combinaison
- malus si aspirine / AINS selon le profil
- malus si association type rhume dans un cas generique
- malus si forme non inalee alors que la demande est inalee
- malus si produit supprime

Exemple sur antipyretic_simple:
- gros bonus pour paracetamol-like
- bonus pour indication fievre
- bonus pour monotherapie
- fort malus pour association rhume
- malus pour aspirine
- malus pour AINS si non prioritaires


7.12. Clinical gate

Apres scoring, il existe un gate clinique.

Exemple:

antipyretic_simple:
- doit etre paracetamol-like
OU
- doit avoir une indication fievre / antipyretique

bronchodilator_inhaled:
- doit etre bronchodilatateur-like
ET
- inaled-like

safety_lookup:
- doit avoir du contenu securite
OU
- de vrais match_reasons

Ce gate est critique:
- il elimine les faux positifs absurdes


7.13. Fallback et preference deterministe

Evolution importante implemente:

Le module ne se contente plus d'attendre une belle reponse LLM.

Il construit un fallback local complet:
- referenced_medicaments
- comparison de secours
- answer de secours
- warnings
- suggestions

Et surtout, pour plusieurs intents metier, le fallback est maintenant
prioritaire ou au minimum pleinement exploitable:

- compare
- explain
- safety
- search metier non generic

Exemple antipyretic_simple:
- le backend peut renvoyer directement une phrase du style
  "Antipyretique usuel: X. Verifier age, poids, grossesse..."

Donc:
- le LLM devient une couche de formulation
- pas le moteur unique de priorisation clinique


7.14. Post-validation

Le modele peut proposer:
- answer
- referenced_medicaments
- comparison
- warnings
- suggestions

Mais apres reponse:
- seuls les medicaments presents dans la shortlist sont acceptes
- la comparison est rejetee si les items ne correspondent pas
- un fallback local remplit les trous si besoin


7.15. Refus du contexte patient

Le chat catalogue ne doit pas faire de pseudo-validation patient.

Donc si la demande ressemble a:
- mon patient
- sa grossesse
- ses traitements
- puis-je lui prescrire

Le module retourne:
- une reponse expliquant que cela releve du futur assistant contextuel
- requires_patient_context = true


7.16. Niveau de maturite

Etat actuel:
- techniquement solide
- pipeline riche
- logique backend bien plus fiable qu'au debut
- mais toujours plus complexe a stabiliser qu'un simple search classique

8. COMPARAISON DES 3 AGENTS
========================================================================

Tableau de synthese:

Agent: hypotheseDiagnostic
- Type: raisonnement clinique
- Entree principale: suivi + examen
- Source verite: dossier patient clinique
- Place du LLM: forte
- Fallback backend: moyen
- Maturite: la plus stable

10. POURQUOI LE SYSTEME A EVOLUE VERS PLUS DE BACKEND
========================================================================

Constat observe pendant les tests:

1. Les providers generalistes sont inegaux.
2. Les outputs JSON varient.
3. Les taches medicamenteuses generiques demandent une priorisation
   clinique stable.
4. La DB medicaments scrapee est semi-structuree et bruyante.

Conclusion technique:

- plus la tache est "deterministe metier", moins il faut laisser le LLM
  choisir seul
- plus la tache est "synthese / reformulation", plus le LLM est utile

Donc le systeme a ete pousse vers:

- backend-first
- retrieval local
- ranking clinique deterministic
- LLM comme couche de formulation / synthese / remplissage quand utile

12. ETAT FINAL RECOMMANDE
========================================================================

Position technique honnete a retenir:

1. hypotheseDiagnostic
- valide pour une V1
- le plus presentable et le plus stable

2. medicationAssistant
- techniquement solide
- utile en interface catalogue
- encore perfectible, mais deja defensable si presente comme
  assistant de recherche / comparaison / resume catalogue

3. ordonnanceRecommendation
- techniquement branche et beaucoup plus robuste qu'au debut
- mais reste le module le plus sensible
- a presenter comme:
  assistant de brouillon therapeutique avec logique backend forte
  et validation medicale imperative

========================================================================
14. CONCLUSION
========================================================================

La logique actuelle des 3 agents n'est pas:
- "on envoie tout au LLM et on espere"

C'est au contraire une architecture mixte:

- contexte structure localement
- retrieval local
- regles backend
- filtrage / exclusion
- scoring metier
- validation de sortie
- fallback deterministe
- LLM uniquement la ou il apporte vraiment de la valeur

Autrement dit:

- hypotheseDiagnostic = AI-clinical reasoning encadre
- ordonnanceRecommendation = therapeutic drafting sous fortes contraintes
- medicationAssistant = assistant catalogue backend-first avec support AI
